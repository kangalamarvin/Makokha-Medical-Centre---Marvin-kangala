# Makokha Medical Centre System - Existing Features (Code-Based Inventory)

This document is generated by reading the current codebase (primarily `app.py`, `config.py`, `utils/*`, `templates/*`, and `static/*`).
It describes the features that already exist and how they work, without changing any application code.

## Quick Map

- Backend: Flask + SQLAlchemy + Jinja templates (`app.py`)
- Realtime: Flask-SocketIO (chat + calling) (`app.py`, `static/js/communication.js`)
- Background jobs: APScheduler (backups, reports, stock alerts, credit reminders, AI dosage agent) (`app.py`)
- Database migrations: Alembic/Flask-Migrate (`migrations/*`)
- Security toolkit: CSRF, rate limiting, WAF, MFA, adaptive auth, SIEM/UBA/DLP/compliance/incident response (`app.py`, `utils/*`)
- Integrations: M-Pesa Daraja, WhatsApp Cloud API, Resend email, AWS S3 (backups and optional uploads) (`app.py`, `utils/*`)

## Roles, Dashboards, and Access Control

### Base roles

The `User` model stores a role (e.g. `admin`, `doctor`, `nurse`, `pharmacist`, `labtech`, `receptionist`, `employee`) and most pages are gated by `@login_required` plus role checks.

### Active role switching (cross-dashboard)

There is a concept of:

- **Base role**: what the user is in the database
- **Active role**: stored in the session for UI + route gating

Supported cross-role switches are allowlist-based (default deny):

- Doctor -> LabTech mode
- Pharmacist -> Receptionist mode

Allowlists are stored in `instance/cross_role_access.json` (`utils/cross_role_access.py`). Routes exist for switching:

- `POST /role/switch/labtech`
- `POST /role/switch/doctor`
- `POST /role/switch/receptionist`
- `POST /role/switch/pharmacist`

When switched, the app intentionally blocks some routes to prevent "mixed permissions" (e.g., doctor endpoints are locked while the doctor is in labtech mode).

## Authentication and Account Security

### Login

Auth routes are implemented as a Flask `Blueprint` mounted at `/auth`:

- `GET|POST /auth/login`
- `GET|POST /auth/verify-mfa`
- `GET|POST /auth/reset-password` (request reset)
- `GET|POST /auth/reset/<token>` (reset with token)
- `GET|POST /auth/verify-email`
- `POST /auth/verify-email/resend`

Important behaviors:

- Login is rate-limited (Flask-Limiter).
- Session fixation prevention: the session is cleared/regenerated before finalizing login.
- Post-login redirect is resolved per-role (admin to admin dashboard, doctor to doctor dashboard, etc.).

### Email verification OTP

There is a verification flow that can send an OTP to the user, store a hash + expiry on the `User` record, and keep "pending verification" context in the session until verified.

### Password reset

Password reset uses a token/nonce and time windows; templates exist under `templates/auth/*`.

### MFA (TOTP) + backup codes

The system supports TOTP MFA (Google Authenticator-compatible) and backup codes:

- Setup: `GET|POST /profile/mfa/setup` and `templates/auth/setup_mfa.html`
- Verify during login: `GET|POST /auth/verify-mfa` and `templates/auth/verify_mfa.html`
- Disable: `POST /profile/mfa/disable`
- Regenerate backup codes: `POST /profile/mfa/regenerate-backup-codes`

Implementation:

- TOTP secrets and backup codes are stored encrypted at rest in the database (`EncryptedType`).
- Feature flags can disable "MFA enforcement" without code changes.

## Security: Phase 1/2/3 System

### Feature flags (persisted)

Security features are toggled by persisted feature flags (`utils/feature_flags.py`) stored at:

- `instance/feature_flags.json`

They are loaded at app startup and copied into `app.config` keys, so runtime code checks only consult `app.config`.

### CSRF protection

CSRF is enabled via Flask-WTF (`CSRFProtect`), and `base.html` exposes a CSRF token in a meta tag for JavaScript:

- `<meta name="csrf-token" content="{{ csrf_token() }}">`

### Rate limiting

Flask-Limiter is initialized using:

- Redis (recommended in production) if `REDIS_URL` or `RATELIMIT_STORAGE_URI` is set
- In-memory storage otherwise (warns in production multi-worker setups)

### Custom WAF (Web Application Firewall)

The app initializes a custom, pattern-based WAF (`utils/custom_waf.py`) that detects and can block:

- SQL injection
- XSS payloads
- path traversal
- command injection
- suspicious headers
- rate-limit-style abuse patterns

It is designed to avoid false positives for clinical free-text (patterns are intentionally conservative).

### Adaptive authentication (risk-based login)

Adaptive auth (`utils/adaptive_auth.py`) calculates a risk score and risk level based on:

- unknown device fingerprint
- unusual IP/location
- unusual login time
- rapid IP change ("impossible travel"-style heuristic)
- suspicious IP patterns
- recent failed login attempts

High-risk logins can force MFA if the user has MFA enabled.

### AI threat detection (Phase 2)

Threat detection (`utils/ai_threat_detection.py`) includes:

- anomaly detection per user profile
- suspicious endpoint pattern detection
- attack sequence heuristics
- scoring + optional auto-blocking (depending on configuration)

### Comprehensive auditing

There is a detailed audit system (`utils/comprehensive_audit.py`) used by critical flows (e.g., login, security alerts, backup actions).

### Phase 3: monitoring and compliance (safe-by-default)

Phase 3 is initialized via `utils/phase3_init.py`:

1. SIEM: structured event pipeline persisted to `instance/siem/` (`utils/siem.py`)
2. UBA: user behavior analytics that reads SIEM telemetry and emits SIEM threat events (`utils/user_behavior_analytics.py`)
3. Compliance checker: posture checks (presence only; no secret values) (`utils/compliance_checker.py`)
4. DLP: response metadata inspection for export-like responses; emits SIEM DLP events; can optionally block (`utils/dlp.py`)
5. Incident response: listens for correlation alerts and can auto-block IPs through the WAF blocklist (`utils/incident_response.py`)

### Database activity monitor

SQLAlchemy engine hooks can emit safe telemetry (no parameters, statement hash only) into SIEM and optionally audit write-ish operations (`utils/db_activity_monitor.py`).

## Encryption and Data Protection

### Field-level encryption (database)

The codebase supports transparent encryption for selected columns via:

- `EncryptedType` (`utils/encrypted_type.py`) using `EncryptionUtils` (`utils/encryption.py`)
- `Config.encrypt_data/decrypt_data` helpers (`config.py`) with legacy key support

Notes:

- Some columns remain plaintext-compatible on non-SQLite DBs (e.g., username/email/role) to preserve indexing/uniqueness in legacy schemas.
- Encryption-at-rest can be toggled for new writes via a feature flag-configured `DB_ENCRYPTION_AT_REST_ENABLED`.

### Private upload encryption (AES-GCM)

User uploads under the private `uploads/` directory can be encrypted using AES-GCM (`utils/upload_encryption.py`).

Key behavior:

- Key preference: `UPLOAD_ENCRYPTION_KEY` -> `FERNET_KEY` -> `BACKUP_ENCRYPTION_KEY` -> `instance/upload_encryption.key`
- File format includes magic bytes (`MMCUP1\n`) so the app can detect encrypted blobs.
- Backward-compatible: non-encrypted legacy files still work.

### Upload persistence + integrity

Uploads are tracked and verified using:

- `UploadedFile` table in the DB (migration: `migrations/versions/001_add_uploaded_files_tracking.py`)
- hashing (SHA-256) + optional backups (`utils/upload_persistence.py`, `utils/upload_management.py`)

There is an admin-only health endpoint:

- `GET /admin/api/uploads/persistence-status`

### Messaging encryption

There are two relevant layers:

1. **Server-side message encryption helper** (Fernet) (`utils/message_encryption.py`), used when enabled by feature flags.
2. **True end-to-end encryption for communication** (WebCrypto, client-side keys):
   - Browser generates RSA keypair, stores private key in `localStorage`.
   - Public key is registered server-side:
     - `POST /api/communication/e2e/register_key`
   - Other users fetch public keys to encrypt messages:
     - `GET /api/communication/e2e/public_key/<user_id>`
   - Sender caches plaintext locally so they can still see their own messages after reload.

This E2E feature is tied to the "Zero Knowledge" feature flag in templates/JS (see `meta[name="mmc-zero-knowledge-enabled"]`).

### Patients

Patient records include:

- OP and IP numbers
- biodata + contacts
- chief complaint + HPI
- department assignment (outpatient)
- admission/discharge workflow (for inpatients)
- AI assistance fields (diagnosis, treatment recommendations, confidence, last updated)

There are append-only sub-record tables (examples):

- `PatientBiodataEntry`
- `PatientChiefComplaintEntry`
- `PatientHPIEntry`

### Patient numbering

The system supports generating patient numbers via:

- `GET /generate_patient_number`

This is typically used by the front desk/doctor flows when registering new patients.

### Wards and beds

Admin manages wards/beds and the system supports:

- Ward definitions (including daily rate)
- Bed inventory and assignments
- Ward stay billing previews and updates

There are API endpoints for ward and bed availability:

- `GET /api/wards`
- `GET /api/wards/<ward_id>/beds`
- `GET /api/wards/<ward_id>/beds/available`

### Lab requests and results

Lab request workflow:

- A doctor creates lab requests for a patient (pending status).
- LabTech dashboard lists pending requests and completes them with results.

LabTech routes:

- `GET /labtech/dashboard`
- `GET /labtech/lab-requests`
- `GET /labtech/lab-request/<request_id>`
- `POST /labtech/lab-request/<request_id>/complete`

### Imaging requests and results

Imaging request workflow mirrors labs:

- Doctor creates imaging request.
- Completion captures results.

### Emergency codes (ward/department broadcast)

Emergency code definitions are in `utils/emergency_codes.py`.
The front-end overlay and speech repetition behavior is implemented in `static/js/emergency-codes.js`.

There are API endpoints under `/api/emergency-codes/*` and an `EmergencyCodeEvent` model for logging activations (explicitly designed to contain no PHI).

## Pharmacy Workflows (Normal + Controlled Drugs)

Pharmacist routes are under `/pharmacist/*`.

### Inventory and dispensing

- Manage drug inventory (remaining quantity, expiry alerts)
- Dispense prescriptions created by doctors
- UI uses a cart system persisted in `localStorage` (see `static/js/main.js`, `static/js/dashboard.js`)

Sales:

- `Sale` / `SaleItem` represent completed sales.

Separate models exist for controlled inventory and prescriptions:

### Dosage index + AI dosage agent

The system supports dosage monographs from:

- a JSON "dosage index book" (searched by normalized drug name)
- an AI dosage generation job system ("dosage dashboard" + background job files under `instance/ai_dosage_jobs/`)

The AI dosage agent can be scheduled and is guarded by enable/disable toggles persisted in `instance/ai_dosage_agent.json`.

### Transactions and receipts

The system uses a `Transaction` ledger table to record money movement (direction IN/OUT), link to source records, and store receipt HTML for audit/reprint.

Receipt HTML is sanitized (Bleach + optional CSS sanitizer) before storing.

### Receipt verification via QR code (public)

Receipts can include a QR that points to:

- `GET /receipt/verify?kind=...&id=...&token=...`

The token is an HMAC derived from `SECRET_KEY` so the verification page can validate that the link is legitimate and the referenced record exists.

## Real-Time Communication System (Chat + Calls)

### Messaging

The communication module supports:

- user list with unread counts
- per-user conversation creation
- message send + delivery/read flags
- typing indicators
- message edits + deletion tracking
- reactions and starring
- reply threading (stores "replied-to message id")
- per-user conversation settings (archived/pinned/muted)
- user blocking (bidirectional checks)

Server API endpoints are under `/api/communication/*` and Socket.IO events handle realtime delivery.

### Voice/video calls (WebRTC signaling)

Calling uses:

- Socket.IO for signaling (offer/answer/ICE candidates)
- browser WebRTC PeerConnection with STUN defaults and optional TURN (`TURN_SERVER_URL`, `TURN_USERNAME`, `TURN_CREDENTIAL`)
- `CallLog` table stores call history and status transitions

Front-end logic lives in `static/js/communication.js`.

### Notifications

Optional push notifications via Firebase Cloud Messaging (FCM) exist (`utils/push_notifications.py`) and can be enabled if FCM env vars are configured.

## Backups and Disaster Recovery

### Backup records and scheduling

Backups are tracked by `BackupRecord` (type: manual/scheduled/disaster_recovery).

Scheduler jobs:

- daily backups (configurable time)
- disaster recovery backups (configurable interval, default 30 minutes)

Backups can be stored:

- locally under the configured backups folder
- and/or uploaded to S3 when configured (`AWS_BACKUP_BUCKET`, AWS credentials)

Backups are encrypted using `BACKUP_ENCRYPTION_KEY` (validated Fernet key in `config.py`).

### Backup access login (separate flow)

There is a dedicated "backup login" subsystem for backup management that uses:

- email registration (admin creates a backup-login user)
- OTP verification
- password setup

Templates exist under `templates/admin/backup_*`.

### Restore

There is a restore route and a doctor-facing "restore last" convenience page:

- `GET|POST /admin/backup` (UI; restore is triggered by a form action on this page)
- `GET|POST /admin/disaster_recovery` (UI)
- `GET /doctor/restore` (doctor UI entry point)

## Uploads: Public vs Private

There are two upload areas:

1. `static/uploads/*`: legacy/public-ish uploads (mostly profile pictures)
2. `uploads/*`: private uploads served only to authenticated users via:
   - `GET /uploads/<path:filename>`

Rules enforced by the private upload server:

- `controlled_prescriptions/*`: admin/pharmacist only
- `profile_pictures/*`: any authenticated user
- directory traversal checks are applied
- if a legacy profile picture exists in `static/uploads`, it can be migrated into the private backend on-demand

## PWA and UI/UX Features

### PWA

The app ships a basic PWA setup:

- `static/manifest.json`
- `static/js/sw.js` service worker (simple cache-first for core assets)

### Vendor asset caching (offline-friendly CDN fallback)

Routes under `/vendor/*` proxy and cache CDN assets (jsdelivr/cdnjs/codejquery) into `instance/vendor_cache/`.
Templates prefer the local `/vendor/...` URL and fall back to upstream CDN on error.

### User settings (client-side)

`static/js/user-settings.js` implements user-controlled preferences stored in `localStorage`:

- timezone (default Africa/Nairobi)
- light/dark mode
- theme color
- background pattern/wallpaper

It dispatches an event (`mmc-settings-changed`) so other JS (notably communication) can react.

### UI customization (server-side)

There is a `UiCustomization` model and admin UI that persists theme choices (e.g., table themes, pharmacist card/button themes).
These are injected into templates as `mmc_ui_customization`.

### Doctor navigation "resume"

`static/js/doctor_resume.js` remembers last doctor URL and scroll position to make returning to long patient pages easier.

## Environment Variables (Names Only)

This project expects a fairly large set of environment variables. The most important groups:

- Flask/security: `SECRET_KEY`, `SECURITY_PASSWORD_SALT`, cookie flags
- Database: `DATABASE_URL`
- Encryption: `FERNET_KEY`, `LEGACY_FERNET_KEYS`, `BACKUP_ENCRYPTION_KEY`, `UPLOAD_ENCRYPTION_KEY`, `MESSAGE_ENCRYPTION_KEY`
- Redis (rate limiting / socketio queue): `REDIS_URL`, `RATELIMIT_STORAGE_URI`, `SOCKETIO_MESSAGE_QUEUE`

- AI (DeepSeek): `DEEPSEEK_API_KEY`, `DEEPSEEK_MODEL`, `DEEPSEEK_VERIFY_ON_STARTUP`, dosage job settings
- WhatsApp Cloud: `WHATSAPP_CLOUD_TOKEN`, `WHATSAPP_PHONE_NUMBER_ID`, `WHATSAPP_API_VERSION`
- WebRTC TURN: `TURN_SERVER_URL`, `TURN_USERNAME`, `TURN_CREDENTIAL`
- M-Pesa Daraja: `MPESA_*` variables (consumer keys, passkey, endpoints, callback URLs)
- AWS S3 backups/uploads: `AWS_BACKUP_BUCKET`, `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `AWS_REGION`, uploads bucket/prefix/SSE settings
